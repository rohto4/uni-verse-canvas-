### 重大な発見: middleware.ts が存在しない

`src/proxy.ts` は Next.js Middleware として実装されているが、
**ファイル名が `proxy.ts` のため Next.js に認識されておらず、一切動いていない。**

- `/admin/*` へのEdgeレベルの保護: **機能していない**
- セッションの自動リフレッシュ (クッキー同期): **機能していない**
- `/login` へのリダイレクト防止: **機能していない**

現在は `(admin)/layout.tsx` のServer Componentガードのみが保護として機能している。
Server Component保護は有効だが、Edgeレベルの保護と比べ以下の点で不足:

- APIルート (`/api/*`) は別途保護が必要
- セッションクッキーのリフレッシュが行われないため、長期間アクセスでセッション切れの可能性

#### TASK-1: `middleware.ts` の作成（`proxy.ts` をリネーム）

**問題**: `proxy.ts` に正しい実装が書かれているが、Next.js Middleware として動作していない。

**対応方針**: `proxy.ts` の内容を `middleware.ts` に移行する。

**実装内容**:

```
src/proxy.ts  →  src/middleware.ts  (リネーム)
```

変更が必要な点:
- エクスポート名を `proxy` → `middleware` に変更（Next.js の規約）
- `config` の `matcher` パターンを確認・調整

**完成後のファイル構成**:
```
src/middleware.ts   # Edge Runtime で動作するミドルウェア
```

**動作確認**:
- [ ] 未ログイン状態で `/admin/dashboard` にアクセス → `/login` にリダイレクトされる
- [ ] ログイン済み状態で `/login` にアクセス → `/admin/dashboard` にリダイレクトされる
- [ ] セッション期限切れ後に `/admin/*` にアクセス → `/login` にリダイレクトされる

**影響ファイル**:
- `src/proxy.ts` → 削除
- `src/middleware.ts` → 新規作成

#### TASK-4: 本番環境変数・OAuth リダイレクト URI の確認

**問題**: ローカル開発では `localhost` に向いているが、本番ドメインに変更する必要がある。

**確認・設定項目**:

Vercel (または本番環境) の環境変数:
```
NEXT_PUBLIC_SUPABASE_URL=       # 本番Supabaseプロジェクト
NEXT_PUBLIC_SUPABASE_ANON_KEY=  # 本番プロジェクトのanon key
SUPABASE_SERVICE_ROLE_KEY=      # 本番プロジェクトのservice role key
```

Supabase Dashboard での設定:
- `Authentication > URL Configuration > Site URL`: 本番ドメインに設定
- `Authentication > URL Configuration > Redirect URLs`: `https://<本番ドメイン>/api/auth/callback` を追加

Google Cloud Console での設定:
- OAuth 2.0 クライアントの「承認済みのリダイレクト URI」に本番URLを追加:
  `https://<本番ドメイン>/api/auth/callback`

**動作確認**:
- [ ] 本番URLでGoogleログインが完了し、`/admin/dashboard` にリダイレクトされる

#### TASK-5: 閲覧数の二重カウント修正

**問題**: `getPostBySlug` 内で `view_count` を更新しているため、
`generateMetadata` の呼び出し時にも加算される（1ページ表示で2回カウントされる可能性）。

**対応方針**: `view-count-plan.md` の A案（クライアントサイドトラッキング）を実装する。

**実装手順**:

1. `src/lib/actions/posts.ts`:
   - `getPostBySlug` から `view_count` 更新ロジックを削除
   - `trackPostView(postId: string)` Server Action を追加
     - `view_count + 1` の更新のみ実行

2. `src/components/posts/PostViewCount.tsx`:
   - マウント時に `trackPostView` を呼ぶ
   - cookie `post_viewed_{postId}` が存在する場合はスキップ
   - cookie を24時間で設定（同一ブラウザの重複を1日1回に抑制）

3. `src/app/(public)/posts/[slug]/page.tsx`:
   - `<PostViewCount postId={post.id} />` を配置

**受け入れ条件**:
- [ ] 記事詳細を表示すると `view_count` が +1 される
- [ ] `generateMetadata` の実行では `view_count` が増加しない
- [ ] 同一ブラウザで24時間以内に再度表示しても `view_count` が増加しない

**影響ファイル**:
- `src/lib/actions/posts.ts`
- `src/components/posts/PostViewCount.tsx`
- `src/app/(public)/posts/[slug]/page.tsx`

## 実装順序と依存関係

```
TASK-1 (middleware)
  │
  ├─→ TASK-4 (環境変数) ← 並行可
  │
TASK-2 (型定義)
  │
  └─→ TASK-3 (エラーページ) ← TASK-1,2 完了後にビルド確認
      │
      └─→ TASK-5 (閲覧数)
          TASK-6 (ログインUX)
          TASK-7 (RLS確認)
```

**推奨実装順**:
1. TASK-2（型定義）→ ビルドエラー解消が最優先
2. TASK-1（middleware）→ セキュリティホールを塞ぐ
3. TASK-4（環境変数確認）→ デプロイ前チェック
4. TASK-3（エラーページ）→ 最低限のUI品質担保
5. TASK-6（ログインUX）→ 軽微なUX改善
6. TASK-5（閲覧数）→ データ品質改善
7. TASK-7（RLS確認）→ セキュリティ最終確認
