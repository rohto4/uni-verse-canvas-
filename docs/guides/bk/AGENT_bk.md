## 定義
docs配下全ての事の中身の事を設計書といいます。
AGENT.mdとDDD.mdの事をコードルールといいます。

## 実装方針
IMP-A.ページの実装がOKとなった段階で必ずあなたが出来る範囲の検証をしてください。
IMP-B.IMP-Aが終わった後には必ず、ほかの実装エージェントと齟齬が発生しない為に設計書を最新化します。
ここで競合が発生するのが致命的なので、工夫して整合性を保つようにしてください。

## コード生成ルール
CODE-001:ソースコードをそのまま読むことはCSS関連以外基本的にないため、エージェントが分かればOKで、冗長なコメントなどは排除してよいです。
ただし、コードを圧縮するために一見わかりづらい処理を実装してしまうのはやめてください。
CODE-002:設計書を生成する際はその他のものを参照しないでも開発が進められるように生成してください。
CODE-003:基本的に設計書をベースに実装します。設計書だけを見ていると実装の齟齬が発生すると思った場合だけ、私に聞いてください。
CODE-004:IMP-Aの実装タスクとIMP-Bの設計書更新タスクは必ずセットで実施してください。私に聞く必要はありません。

## エラー対応原則
・エラーが頻発する、またはエラーが多い場合のテスト方法として、
試行回数の節約を心がけてください。
・全てのエラーについて分析し、分析に対して成果物・修正の山が出てきます。
これを私がまとめて承認します。（承認が不要なフローになっていればスキップ）
・これを1往復して治りきらなかった場合、temp-error-report.mdを引き出してきてください。

・現在のエラーをこのファイルに一覧化し、やはり全てのエラーに対して、修正案を出してから許可を取るようにお願いします。
ただし、無許可で修正が走らせられるのが理想です。

## 推奨開発フロー

本プロジェクトでは、開発時(`dev`)と本番ビルド時(`build`)のエラー検知の差異をなくし、開発効率を上げるため、以下の2つのターミナルを並行して使用することを推奨します。

### ターミナル1: 開発サーバーの起動
```bash
npm run dev
```
Next.jsの開発サーバーを起動し、高速リフレッシュによるUIの確認を行います。

### ターミナル2: 型チェックの常時監視
```bash
npm run type-check
```
TypeScriptの型チェッカーを監視モードで起動します。これにより、ファイルを保存するたびに、ビルド時と同じ厳密な型チェックがリアルタイムで実行され、エラーを即座に検知できます。
